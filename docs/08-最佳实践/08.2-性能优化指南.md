## ⚠️ 性能优化优先级

> **优化策略建议**:
>
> - ✅ **先测量再优化**: 使用`opa bench`和profiling确定瓶颈
> - ✅ **80/20法则**: 优先优化最热路径（通常20%代码占80%性能）
> - ✅ **数据结构优先**: 索引优化通常比算法优化效果更显著
> - ✅ **验证优化效果**: 每次优化后重新测量，确认改进
> - ⚠️ 过度优化可能降低可读性，权衡tradeoff
> - ⚠️ 微优化（如减少一次循环）通常不如结构性优化
>
> **典型性能基准**:
>
> - Sidecar模式: P99 < 5ms, QPS > 10K
> - 集中式: P99 < 10ms, QPS > 50K  
> - WASM: P99 < 2ms, QPS > 100K
> - **Batch API**: 吞吐量提升 **18x** vs 单个查询 (Gusto 2024)
>

## 📝 English Summary

This is the definitive guide for optimizing OPA policy performance in production systems.

**Key Topics**:

- **Measurement**: Profiling with `opa bench`, `--profile` flag, metrics analysis
- **Indexing**: Hash table lookup (O(1)) vs. array iteration (O(n))
- **Query Optimization**: Query ordering, early termination, caching strategies
- **Data Structures**: Sets vs. arrays, object keys vs. values
- **Architecture**: Sidecar vs. centralized, WASM compilation, partial evaluation
- **Batch API**: Bulk query optimization for high-throughput scenarios
- **Storage Optimization**: `--optimize-store-for-read-speed` for read-heavy workloads
- **Kubernetes Tuning**: GOMAXPROCS and CPU core optimization

**Performance Targets**: P99 < 10ms, Throughput > 10k req/s (varies by deployment)

**Target Audience**: Performance engineers, DevOps, production OPA operators.

## 2. 策略层优化

### 2.1 避免反模式

**❌ 反模式1: 全表扫描**:

```rego
# 慢: O(n)
user_by_email(email) := user if {
    some user in data.users
    user.email == email
}

# 快: O(1)
user_by_email(email) := data.users_by_email[email]
```

**❌ 反模式2: 笛卡尔积**:

```rego
# 慢: O(n²)
conflicts := [[u1, u2] |
    some u1 in data.users
    some u2 in data.users
    u1.team == u2.team
]

# 快: O(n)
by_team := {team: users |
    some user in data.users
    team := user.team
    users := [u | some u in data.users; u.team == team]
}
```

**❌ 反模式3: 重复计算**:

```rego
# 慢: 多次计算
allow if {
    count(input.permissions) > 5
    count(input.permissions) < 20
}

# 快: 缓存结果
perm_count := count(input.permissions)
allow if {
    perm_count > 5
    perm_count < 20
}
```

### 2.2 最佳实践

**✅ 使用集合操作**:

```rego
# 集合交集
allowed := user_permissions & required_permissions

# 集合包含
allow if {
    "admin" in input.user.roles
}
```

**✅ 提前退出**:

```rego
allow if {
    # 快速检查在前
    input.method == "GET"
    
    # 昂贵操作在后
    user := data.users[input.user_id]
    check_permissions(user)
}
```

**✅ 索引访问**:

```rego
# 直接索引
user := data.users[input.user_id]
role := data.roles[user.role_id]

# 避免搜索
# ❌ some user in data.users; user.id == input.user_id
```

### 5.2 连接池

**Go应用**：

```go
// 复用OPA实例
var opaPool = sync.Pool{
    New: func() interface{} {
        opa, _ := rego.New(
            rego.Query("data.authz.allow"),
            rego.Load([]string{"policy.rego"}, nil),
        ).PrepareForEval(context.Background())
        return opa
    },
}

func authorize(ctx context.Context, input map[string]interface{}) (bool, error) {
    pq := opaPool.Get().(rego.PreparedEvalQuery)
    defer opaPool.Put(pq)
    
    rs, err := pq.Eval(ctx, rego.EvalInput(input))
    if err != nil {
        return false, err
    }
    return rs.Allowed(), nil
}
```

### 5.3 Prepared Queries 预解析查询

**预编译查询优化**：

```go
// 推荐: 使用PrepareForEval预编译查询
ctx := context.Background()

// 一次性准备查询（启动时）
query, err := rego.New(
    rego.Query("data.authz.allow"),
    rego.Load([]string{"policy.rego"}, nil),
).PrepareForEval(ctx)

if err != nil {
    log.Fatal(err)
}

// 多次执行（运行时）- 每次调用避免解析和编译开销
results, err := query.Eval(ctx, rego.EvalInput(input1))
results, err = query.Eval(ctx, rego.EvalInput(input2))
// ...
```

**优势**：

```text
✅ 避免每次查询的解析和编译开销
✅ 可在多个goroutine中安全共享
✅ 显著提升高频查询场景性能
✅ 减少GC压力（减少临时对象分配）

性能对比:
  无Prepare: 每次查询 ~5-10ms (包含编译)
  有Prepare: 每次查询 ~0.1-0.5ms (仅执行)
  提升: 10-100x
```

## 6. 存储优化

### 6.1 内存存储读取优化

**`--optimize-store-for-read-speed` 标志** (OPA v0.50+):

```bash
# 启用存储读取优化
opa run --optimize-store-for-read-speed \
    --server \
    --bundle policy.bundle

# 基准测试时启用
opa bench --optimize-store-for-read-speed \
    -d data.json \
    -i input.json \
    'data.authz.allow'

# 评估时启用
opa eval --optimize-store-for-read-speed \
    -d data.json \
    -i input.json \
    'data.authz.allow'
```

**工作原理**：

```text
正常操作:
  存储数据 → 读取时转换为AST → 策略评估
  └── 每次读取都需转换，耗时且增加GC压力

优化后:
  存储数据 → 预计算AST表示 → 策略评估直接使用AST
  └── 读取时无需转换，直接复用预计算AST

权衡:
  ✅ 读取速度显著提升 (20-50% 延迟减少)
  ✅ 内存使用更稳定（减少评估时内存峰值）
  ✅ 数据在多次评估间共享
  
  ⚠️ 内存占用增加 (AST比原始数据占用更多空间)
  ⚠️ 写入操作变慢 (启动、bundle加载、数据更新)
  ⚠️ 不适合写密集型场景
```

**适用场景**：

```text
✅ 读密集型工作负载 (推荐)
✅ 大数据集频繁查询
✅ 延迟敏感型应用
❌ 写密集型工作负载
❌ 内存受限环境
❌ 频繁更新数据的场景
```

**性能基准**：

```text
场景: 100,000条权限数据查询

无优化:
  P50: 2.5ms
  P99: 8ms
  内存峰值: 高（每次评估分配）

有优化:
  P50: 1.2ms (52% ↓)
  P99: 4ms (50% ↓)
  内存峰值: 低（预分配共享）
  启动时间: +30%
  内存占用: +25%
```

### 7.3 Kubernetes性能优化

**GOMAXPROCS与CPU核心数配置** (参考Gusto 2024实践):

```text
问题背景:
  Kubernetes CPU限制: 750m (0.75核)
  Go默认GOMAXPROCS: 8 (检测主机所有核心)
  结果: 8个线程竞争0.75核，性能极差

根本原因:
  Go根据操作系统报告的核心数设置GOMAXPROCS
  Kubernetes CPU限制不会反映在/proc/cpuinfo中
  导致Go创建过多线程，引发上下文切换开销
```

**优化策略**：

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: opa
          image: openpolicyagent/opa:latest
          resources:
            requests:
              cpu: "500m"
            limits:
              cpu: "3000m"  # 给足CPU避免节流
          env:
            # 方式1: 手动设置GOMAXPROCS匹配CPU限制
            - name: GOMAXPROCS
              value: "3"  # 匹配3000m限制
            
            # 方式2: 使用automaxprocs自动检测
            # 需要引入 github.com/uber-go/automaxprocs
          
          # 同节点亲和性（与业务Pod同节点）
          affinity:
            podAffinity:
              preferredDuringSchedulingIgnoredDuringExecution:
                - labelSelector:
                    matchLabels:
                      app: my-service
                  topologyKey: kubernetes.io/hostname
```

**Gusto生产配置参考**：

```text
最终优化配置:
  Kubernetes CPU Limit: 3000m (3核)
  GOMAXPROCS: 3
  
性能结果:
  Batch API平均延迟: 400ms → 45ms (89% ↓)
  单请求延迟: 3ms → 1.5ms (50% ↓)
  吞吐量: 18x 提升

测试方法:
  使用opa bench测试不同CPU限制(1-4核)
  测试不同GOMAXPROCS值(1-6)
  共18种组合找出最优配置
```

**资源限制最佳实践**：

```text
通用规则:
  1. GOMAXPROCS 通常应与CPU核心数匹配
  2. CPU Limit 应 ≥ GOMAXPROCS (避免节流)
  3. 留有余量应对突发流量

推荐配置:
  ┌─────────────────┬──────────────┬─────────────┐
  │ 场景             │ CPU Limit    │ GOMAXPROCS  │
  ├─────────────────┼──────────────┼─────────────┤
  │ 小型服务         │ 1000m        │ 1           │
  │ 中型服务         │ 2000m        │ 2           │
  │ 大型服务         │ 3000-4000m   │ 3-4         │
  │ 高性能批处理     │ 4000m+       │ 4           │
  └─────────────────┴──────────────┴─────────────┘
```

## 8. 索引与查询优化

### 8.1 规则索引最佳实践

**索引工作原理**：

```rego
package indexed

# 这些规则会被OPA自动索引
allow if {
    some user
    input.method == "GET"              # 索引键1
    input.path = ["accounts", user]    # 索引键2
    input.user == user                 # 索引键3
}

allow if {
    input.method == "GET"              # 索引键1
    input.path == ["accounts", "report"]
    roles[input.user][_] == "admin"    # 索引查找
}

allow if {
    input.method == "POST"             # 索引键1
    input.path == ["accounts"]
    roles[input.user][_] == "admin"
}
```

**索引友好的表达式**：

| 表达式 | 索引支持 | 说明 |
|--------|----------|------|
| `input.x == "foo"` | ✅ 是 | 简单等式 |
| `input.x.y == "bar"` | ✅ 是 | 嵌套引用 |
| `input.x == ["foo", i]` | ✅ 是 | 包含变量数组 |
| `input.x[i] == "foo"` | ❌ 否 | 引用包含变量 |
| `input.x[input.y] == "foo"` | ❌ 否 | 嵌套引用含变量 |
| `glob.match("foo:*:bar", [:], input.x)` | ✅ 是 | 标准glob模式 |
| `glob.match("foo:**:bar", [:], input.x)` | ❌ 否 | 超级glob ** |

### 8.2 部分求值优化新建议

**理解求值阶段**：

```rego
# 输入:
allow if {
    input.method == "GET"
    input.path == ["salary", input.user]
}

# 部分求值后 (静态数据已知):
allow if {
    input.method == "GET"
    input.path == ["salary", input.user]
}
# 仅剩input依赖，运行时求值极快
```

**部分求值优化级别**：

```bash
# -O=1: 推荐级别，不依赖unknown的规则完全求值
opa build -O=1 --bundle policy/

# -O=2: 激进级别，可能内联依赖unknown的规则
opa build -O=2 --bundle policy/
```

**部分求值最佳实践** (2024更新)：

```text
1. 分离静态数据和动态输入
   - 将不经常变化的数据放入data
   - 将每次请求变化的放入input

2. 优化Bundle构建时机
   - 数据变化时重新构建
   - 使用CI/CD自动化

3. 监控优化效果
   - 比较优化前后的延迟
   - 注意内存占用变化

4. 谨慎使用-O=2
   - 充分测试策略行为一致性
   - 关注边界情况
```

### 8.3 理解索引（Comprehension Indexing）

**避免O(n²)复杂度**：

```rego
# 场景: 统计每个接口暴露的端口数
# 不使用索引: O(n²)
deny contains msg if {
    some i
    intf := input.exposed[i].interface
    ports := [port | 
        some j
        input.exposed[j].interface == intf  # 每次重新计算
        port := input.exposed[j].port
    ]
    count(ports) > 100
    msg := sprintf("interface '%v' exposes too many ports", [intf])
}

# 使用索引: O(n)
# OPA会自动缓存comprehension结果
deny contains msg if {
    some intf, ports in exposed_ports_by_interface
    count(ports) > 100
    msg := sprintf("interface '%v' exposes too many ports", [intf])
}

exposed_ports_by_interface := {intf: ports |
    some i
    intf := input.exposed[i].interface
    ports := [port |
        some j
        input.exposed[j].interface == intf
        port := input.exposed[j].port
    ]
}
```

**索引条件**（必须全部满足）：

```text
✅ Comprehension出现在赋值或统一语句中
✅ 表达式不包含with语句
✅ 表达式未被否定
✅ Comprehension body独立安全
✅ 关闭至少一个外部变量
❌ 外部变量不出现在引用操作数或walk()调用中
❌ 外部变量不在嵌套comprehension中
```

## 10. 实战案例

### 10.1 大规模RBAC

**场景**: 10,000用户, 1,000角色, 100,000权限

**优化前**：

```rego
# P99: 50ms
allow if {
    some role in input.user.roles
    some perm in data.role_permissions[role]
    perm.resource == input.resource
    perm.action == input.action
}
```

**优化后**：

```rego
# 1. 预计算用户权限
user_perms := {p |
    some role in input.user.roles
    some p in data.role_permissions[role]
}

# 2. 使用集合操作
required := {input.resource, input.action}
allow if {
    some perm in user_perms
    {perm.resource, perm.action} == required
}

# P99: 0.5ms (100x提升)
```

### 10.2 实时日志过滤

**场景**: 过滤1M条日志

**优化策略**：

```rego
# 1. 布隆过滤器预筛选
suspicious_ips := bloom_filter_from_set(data.blacklist)

filtered_logs := [log |
    some log in input.logs
    bloom_filter.may_contain(suspicious_ips, log.ip)  # 快速否定
    exact_check(log)  # 精确检查

# 2. 位图索引
logs_by_level := {
    "ERROR": roaring.bitmap([1, 5, 10, ...]),

error_logs := [input.logs[i] |
    some i in roaring.to_array(logs_by_level["ERROR"])
]

# 性能: 5s → 50ms (100x提升)
```

## 12. 快速诊断

**性能问题决策树**：

```text
性能慢？
├─ 延迟高？
│  ├─ 策略复杂？→ 简化规则，优化算法
│  ├─ 数据大？→ 索引，分片，部分求值
│  ├─ 存储转换慢？→ --optimize-store-for-read-speed
│  ├─ GOMAXPROCS不匹配？→ 调整以匹配CPU限制
│  └─ 网络慢？→ Sidecar模式，WASM
│
├─ 吞吐低？
│  ├─ 单请求瓶颈？→ 使用Batch API
│  ├─ CPU瓶颈？→ 增加副本，优化算法
│  ├─ 内存不足？→ 优化数据结构，分片
│  └─ GC频繁？→ 调整GOGC，增加内存
│
├─ 批处理慢？
│  ├─ 未使用common_input？→ 提取公共数据
│  ├─ 请求过大？→ 减小批次大小
│  └─ 个别慢请求拖累？→ 分析决策指标，隔离问题
│
└─ 内存高？
   ├─ 数据大？→ 分片，按需加载
   ├─ 缓存大？→ 调整缓存大小
   ├─ 存储优化开启？→ 权衡读写比例
   └─ 泄漏？→ Profiling，修复代码
```
